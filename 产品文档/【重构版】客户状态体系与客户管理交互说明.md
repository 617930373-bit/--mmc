## 二、重构目标

本次重构的核心不是单纯“加几个状态”，而是重新梳理客户在线索进入、分配、维护、跟进失败、公海再利用之间的完整流转链路。

原系统中的“公海”混合了承接新线索待分配的客户，以及销售主动放弃/系统回收的历史客户，导致以下问题：

- 新客和老客混在一起，影响新线索分配效率。
- 销售跟不动的客户长期沉淀，缺少再次流转机制。
- 公海客户仍然按“分配”思路处理，无法体现“主动领取”的业务模式。
- 页面状态、跟进动作、业务含义混用，导致用户理解成本高，数据看板也难以准确统计。

因此，本次需要将“老公海”拆分为两类资源池：

- `待分配`：承接新线索或老客户新线索再次进入后、需要重新进入分配流程的客户。
- `公海中`：承接销售主动放弃、战败/无效释放、超期规则回收的客户，仅允许他人主动领取。

## 四、核心概念定义

| 概念 | 定义 |
|---|---|
| 分配 | 新线索客户分给某个销售/评估师，无论自动分配还是手动分配，统一称为分配 |
| 转移 | 已有归属人的客户，由有权限人员转给其他人员 |
| 领取 | 销售/评估师从公海中手动获取客户 |
| 待分配 | 待进入分配流程的客户池，只允许分配，不允许领取 |
| 公海中 | 已被放弃或被规则释放的客户池，只允许领取，不允许分配 |
| 新客待维护 | 新线索完成分配后的首咨待维护状态 |
| 待维护 | 客户从公海领取后的待维护状态 |
| 跟进中 | 已开始实际维护的客户状态 |
| 邀约/到店 | 不再作为独立客户状态，而是跟进行为或动作标记；客户状态仍属于 `跟进中` |

## 五、状态设计总原则

### 5.1 业务原则

- 新客走分配逻辑。
- 公海走领取逻辑。
- 新线索优先保障快速分配和转化。
- 无法转化或超期未转化客户，应进入可再次利用的公海。
- 状态要服务于业务流转，也要服务于后续管理看板统计。

### 5.2 本次确认的关键口径

以下 4 条为已确认口径，应作为最终规则执行：

1. `新客待维护` 和 `待维护` 是两个不同状态，不可合并。
2. 客户再次有新线索进入时，仅 `公海中` 客户会重新进入 `待分配`；其他状态均保持原状态不变，只新增轨迹，并将下次跟进时间更新为 `线索流入时间 + 12 小时`。
3. `邀约` 和 `到店` 不再作为独立客户状态，而是跟进动作标记；对应客户状态统一视为 `跟进中`。
4. 卖车业务与买车业务采用同一套主状态框架，仅将“订单中”替换为“采购中”。

## 六、客户状态体系

### 6.1 买车业务客户状态

| 阶段 | 状态 | 是否有归属人 | 是否可掉入公海 | 进入条件 | 说明 |
|---|---|---|---|---|---|
| 新客阶段 | 待分配 | 否 | 否 | 新线索进入且未分配；客户创建时未选择归属；公海客户再次有新线索进入 | 只允许分配，不允许领取 |
| 新客阶段 | 新客待维护 | 是 | 否 | 待分配客户完成分配后进入 | 表示首咨待维护 |
| 新客阶段 | 待维护 | 是 | 是 | 客户从公海被领取后进入 | 表示公海领取后的待维护 |
| 跟进阶段 | 跟进中 | 是 | 是 | 新客待维护/待维护开始跟进后进入；员工自己创建并归属给自己 | 邀约、到店均属于该状态下的动作标记 |
| 接近成功阶段 | 订单中 | 是 | 是 | 产生销单后进入 | 原“买车预定”更名为“订单中” |
| 成功阶段 | 成交 | 是 | 否 | 订单全部交款完成；或管理员手动标记成交 | 成交后不再进入待分配 |
| 跟进失败阶段 | 公海中 | 否 | 否 | 手动丢公海；跟进时选择战败/无效；超期规则回收 | 只允许领取，不允许分配 |
| 特殊阶段 | 废弃 | 可无归属 | 否 | 管理员手动标记 | 终态，不重新分配 |

| 跟进阶段 | 跟进中 | 是 | 是 | 新客待维护/待维护开始跟进后进入；员工自己创建并归属给自己 | 与买车一致 |

| 特殊阶段 | 废弃 | 可无归属 | 否 | 管理员手动标记 | 终态，不重新分配 |

### 6.3 意向等级

买车、卖车统一支持以下意向等级：

- `H`
- `A`
- `B`
- `C`
- `N`
- `战败`
- `无效`
- `成交`

补充规则：

- `成交` 既是客户状态，也是意向等级，但只有客户状态为 `成交` 时才允许选择 `成交` 意向等级。
- 战败、无效本质上会触发客户释放到公海。

## 七、状态流转规则

### 7.1 新线索进入

#### 场景一：系统中不存在该客户

- 新线索进入后，客户状态为 `待分配`。
- 完成分配后，客户状态变为 `新客待维护`。

#### 场景二：系统中已存在该客户

- 若当前状态为 `公海中`，则该客户重新进入 `待分配`，并按最新进线来源走分配流程。
- 若当前状态不是 `公海中`，则客户状态保持不变。
- 此时系统只新增一条线索轨迹，并将下次跟进时间更新为 `线索流入时间 + 12 小时`。

### 7.2 分配

- `待分配` 客户只能执行分配。
- 分配后客户默认进入 `新客待维护`。
- 已有归属人的客户不再出现“分配”能力，应改为“转移”。

### 7.3 领取

- `公海中` 客户只能执行领取。
- 领取后客户默认进入 `待维护`。
- `新客待维护` 与 `待维护` 需要在取数时可区分来源。

### 7.4 跟进

- `新客待维护` 或 `待维护` 点击跟进后，进入 `跟进中`。
- `邀约`、`到店` 不再改变客户主状态，只记录为跟进行为标记。
- 员工自己创建客户，或通过销单入口创建客户，默认直接进入 `跟进中`，避免被新客回收逻辑误回收。

### 7.5 丢公海

以下场景都应将客户释放到 `公海中`：

- 员工主动放弃客户
- 跟进结果选择 `公海中`
- 意向等级选择 `战败`
- 意向等级选择 `无效`
- 后续规则任务判定超期回收

统一规则：

- 丢公海必须伴随一次跟进记录。
- 丢公海后不再保留下次跟进时间。
- 丢公海时需记录具体原因与跟进内容。

### 7.6 接近成功与成交

买车业务：

- 产生销单后进入 `订单中`。
- 订单全部交款后进入 `成交`。

卖车业务：

- 发起采购后进入 `采购中`。
- 系统完成收车后进入 `成交`。

补充规则：

- `订单中`、`采购中` 是系统状态，用户不可随意手动变更。
- `订单中`、`采购中`、`成交` 的跟进页不再允许用户修改客户状态和意向等级，只允许补充跟进内容。

## 八、存量数据上线迁移规则

### 8.1 买车业务迁移规则

- 原“新建”且无归属的客户，统一迁移为 `待分配`。
- 原“新建”且有归属的客户，统一迁移为 `跟进中`。
- 原“邀约”或“到店”且有归属的客户，统一迁移为 `跟进中`。
- 原“战败”或“无效”且有归属的客户，清空归属并迁移为 `公海中`。

### 8.2 卖车业务迁移规则

| 原状态 | 是否有归属 | 新状态 | 备注 |
|---|---|---|---|
| 待评估 | 无归属 | 待分配 | 进入新分配池 |
| 待评估 | 有归属 | 跟进中 | 视为已在维护 |
| 评估中 | 全部 | 采购中 | 状态重命名 |
| 成交 | 全部 | 成交 | 保持不变 |
| 战败 | 全部 | 公海中 | 意向等级保留为战败 |
| 无效 | 全部 | 公海中 | 意向等级保留为无效 |

### 9.3 待分配列表页

业务规则：

- 待分配只允许分配，不允许领取
- 支持单个分配、批量分配
- 买车待分配页去掉卖车意向字段
- 卖车待分配页去掉买车意向字段
- PC 端无需新增独立页面时，可复用线索管理中的客户模式作为待分配页

### 9.4 公海列表页

公海列表页与待分配页完全区分。

业务规则：

- 公海只允许领取，不允许分配
- 支持批量领取
- 不展示领取记录
- 不展示“新增”能力
- 每页最多展示 20 条
- 每人每日领取限额暂定 100 条，支持配置
- 公海页面必须先选择门店，禁止“全部门店”
- 进入页面时记忆上次选择的门店

## 十、跟进页调整

### 10.1 通用规则

- 取消“不跟进”选项
- 必须填写跟进结果
- 选择意向等级后，自动带出对应的下次跟进时间
- 战败、无效、公海中均不需要下次跟进时间
- 支持上传“其他图片资料”，单张 5MB 以内，最多 10 张

### 10.2 丢公海、战败、无效联动

- 选择 `战败` 时，客户状态默认切为 `丢公海`
- 选择 `无效` 时，客户状态默认切为 `丢公海`
- 战败/无效理由为“其他”时，跟进内容必填
- 战败/无效理由非“其他”时，跟进内容可不必填
- 无论之前是什么意向等级，只要客户状态选择为 `丢公海`，下次跟进时间都要清空

### 10.3 买车跟进页

- 支持强化填写意向车系、预算价格，但不强制必填
- 沟通方式选择“到店”时，需填写到店时间、离店时间
- 其他沟通方式不展示到店相关字段
- 当买车意向都已结束时，可进入快速跟进模式，仅填写跟进内容
- 当无买车意向时，提示先新增买车意向再跟进

### 10.4 卖车跟进页

- 不填写预算
- 卖车跟进针对卖车客户，不展示买车意向相关交互
- 电话、微信、短信联系时，不需要选择邀约、预计到店时间等字段
- 重新从公海领取的客户，不应自动带出上一次的战败/无效意向等级，默认应为空
- 当卖车意向均已结束时，可快速跟进，仅填写内容
- 当无卖车意向时，提示先新增卖车意向再跟进

## 十一、详情页调整

### 11.1 通用规则

- 有归属客户不再显示“分配”，仅支持“转移”
- 无归属客户不能转移；待分配客户走分配，公海客户走领取

### 11.2 买车详情页

- 隐藏“结束意向”操作
- 成交、战败、无效一律通过跟进页处理

### 11.3 卖车详情页

- 保留“结束意向”操作
- 但卖车结束意向仅作用于单个卖车意向，不直接决定客户主状态

### 11.4 放弃按钮调整

- 移动端详情页中，“放弃”文案改为 `放弃-丢公海`
- 默认选中客户状态为 `丢公海`
- PC 端详情页可去掉单独“放弃”按钮，统一通过跟进完成

## 十二、状态轨迹记录要求

丢公海后的轨迹需要完整保留，至少包括：

- 操作名称：如“丢公海”
- 操作人
- 操作时间
- 跟进内容
- 沟通方式
- 意向等级
- 战败/无效原因

此外，公海入库原因需要补充枚举值：

- 员工放弃
- 线索战败
- 线索无效

## 十三、页面展示简化要求

由于新的状态体系已经替代原有进度条逻辑，因此：

- 取消客户进度条
- 直接展示当前状态值
- 移动端意向详情中的进度条、意向等级展示也一并简化

## 十四、实施重点总结

本次重构的本质可以概括为 5 句话：

1. `待分配` 和 `公海中` 必须彻底拆开，前者走分配，后者走领取。
2. `新客待维护` 和 `待维护` 必须区分，因为两者来源不同，后续回收逻辑也不同。
3. `邀约`、`到店` 不再作为客户状态，而是 `跟进中` 下的动作标记。
4. 战败、无效、主动放弃，本质上都属于“释放客户到公海”的动作。
5. 客户状态设计要同时支持业务流转、页面理解和管理看板统计。

## 十五、后续能力建议

原始需求中已经提到但本次建议放后续版本推进的能力包括：

- 新客待维护的超期回收再分配
- 跟进中客户的维护周期控制
- 公海自动回收规则设定
- 销售库容控制
- 销转等级控制接新、公海领取、未成交客户占用数
- 更细粒度的线索来源分配策略

这些能力与本次状态重构强关联，但建议拆分为后续专题需求。
